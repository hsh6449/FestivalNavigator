# FestivalNavigator Working Rules

이 저장소에서 작업할 때는 구현을 바로 시작하기 전에, 요청 유형에 맞는 템플릿으로 먼저 정리한다.

## 1. 구현 작업 템플릿

아래 작업을 바로 구현하지 말고, 먼저 작업 계획으로 정리해줘.

목표:
-

- 새 라이브러리 추가 금지
- 관련 파일만 최소 수정
- 정책/보안/DB 스키마 변경은 수정하지 말고 질문으로 돌릴 것

완료 조건:
- 수정 대상 파일 후보
- 구현 계획
- 필요한 테스트
- 사람이 승인해야 할 결정

### 출력 규칙

- 구현 전에 반드시 계획부터 제시한다.
- 계획은 현재 저장소 구조와 실제 파일 후보를 기준으로 작성한다.
- 범위가 커지면 단계별로 쪼개고, 즉시 구현 가능한 최소 단위도 함께 적는다.
- 정책, 권한, 보안, DB 스키마 변경이 필요해 보이면 구현으로 넘기지 말고 승인 항목으로 분리한다.

## 2. 버그 수정 템플릿

아래 버그를 수정해줘. 단, 바로 고치지 말고 재현과 실패 테스트를 먼저 제안해줘.

증상:
-

- 관련 화면/기능:
- 관련 파일 후보:

제약:
- 최소 수정
- 새 라이브러리 추가 금지
- 관련 없는 리팩터링 금지

완료 조건:
- 재현 경로
- 실패 테스트 1개
- 수정 diff 요약
- 실행한 검증 명령
- 수동 확인 순서

### 출력 규칙

- 바로 수정안부터 내지 말고, 먼저 재현 경로와 실패 테스트를 제시한다.
- 실패 테스트는 실제 버그를 잡아낼 수 있는 가장 작은 범위로 제안한다.
- 수정 후에는 변경 파일과 diff 요약을 짧게 정리한다.
- 검증 명령과 수동 확인 순서를 반드시 남긴다.

## 3. 코드 리뷰 템플릿

이 변경사항을 리뷰해줘.

집중할 것:
- 회귀 위험
- 빠진 테스트
- 보안/권한 문제
- 과한 변경
- 되돌리기 어려운 변경
- 사람이 직접 승인해야 할 항목

출력:
- Critical / Major / Minor로 분류
- 파일별 코멘트
- 반드시 확인할 테스트
- merge 전에 필요한 조치

### 출력 규칙

- 리뷰는 요약보다 이슈를 우선한다.
- 이슈가 없으면 명시적으로 없다고 말하고, 남아 있는 테스트 갭이나 관찰 포인트를 적는다.
- 사람이 승인해야 할 결정은 별도로 분리한다.
- 관련 없는 스타일 논쟁보다 회귀, 권한, 데이터 안정성, 운영 리스크를 우선한다.

## 4. 공통 원칙

- 새 라이브러리 추가는 기본적으로 금지한다.
- 관련 없는 리팩터링은 하지 않는다.
- 수정은 최소 범위로 시작한다.
- 테스트 가능하면 자동 검증을 먼저 돌리고, 부족하면 수동 확인 순서를 적는다.
- seed 데이터, mock 데이터, DB fallback이 함께 있는 구조에서는 어느 경로가 실제로 사용되는지 먼저 확인한다.
